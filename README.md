# easy

## 简介

`easy` 是一个面向**买球站开发场景**沉淀的工具集合与实践仓库，聚焦于日常开发中高频、重复、容易出错的环节，帮助团队快速搭建更稳定、更规范、更易维护的工程体系。无论是前端页面组织、接口请求封装、通用工具函数管理，还是项目结构约定、协作流程与上线前检查，`easy` 都希望提供一套“开箱可参考、落地可复用、迭代可扩展”的方案。

在实际业务推进中，买球站类项目往往具有节奏快、页面多、活动频繁、接口联动复杂等特点。如果缺少统一的工程约束和公共能力，项目很容易在多人协作中出现代码风格不一致、模块重复开发、联调效率低、上线风险高等问题。`easy` 的目标，就是把这些常见问题前置为可复制的方法论与可直接使用的工具模块。⚙️

它并不局限于某一种技术栈，而更强调“实用主义”——优先解决问题、提升效率，并为团队沉淀长期可持续的研发资产。

## 特点

- **聚焦买球站开发场景**  
  针对活动页、专题页、业务中台、接口联调等常见需求进行整理，更贴近真实项目落地。

- **工具与规范并重**  
  不只是提供脚本和函数库，也包含工程目录建议、命名规范、提交规范与协作建议。

- **支持快速复用**  
  将高频逻辑抽离成通用模块，减少重复造轮子，让新项目启动更轻量。

- **强调可维护性**  
  通过统一封装和结构设计，降低后期修改成本，方便多人协作与长期演进。

- **适合团队沉淀最佳实践**  
  可作为内部脚手架、公共仓库或项目模板的基础，为后续扩展提供稳定起点。

- **兼容渐进式接入**  
  不要求一次性重构全部项目，可以按模块逐步引入，在存量项目中也能平滑使用。

- **面向效率优化**  
  从开发、联调、测试到发布，尽可能缩短重复劳动时间，提高整体交付效率 🚀

## 快速开始

根据你的团队技术栈，可以将 `easy` 作为工具仓库、模板仓库或公共模块仓库引入。一个常见的使用方式如下：

```bash
# 克隆仓库
git clone https://github.com/your-org/easy.git

# 进入目录
cd easy

# 安装依赖
npm install

# 本地开发
npm run dev

# 构建产物
npm run build
```

如果你希望将其中的工具函数、请求封装或规范配置迁移到现有项目，建议优先从以下部分开始：

1. 公共工具函数  
2. HTTP 请求层封装  
3. ESLint / Prettier / Commit 规范  
4. 环境变量与构建配置  
5. 常用业务组件或页面模板

对于新项目，推荐直接基于 `easy` 初始化一套目录结构，再按具体业务进行裁剪。这样可以更快地统一团队习惯，并减少启动阶段的决策成本。

## 使用说明

`easy` 更适合作为一个“工程能力底座”来使用。实际接入时，建议从模块化思路出发，而不是一次性全量复制。

### 1. 工具库复用

将日期处理、参数解析、防抖节流、格式化、存储封装等通用逻辑统一放入 `utils` 目录，通过明确的导出入口进行管理。这样可以避免各业务模块各写一套，减少隐藏 bug 的来源。

### 2. 请求层统一

对于买球站类项目，接口调用频繁且场景多变，建议统一处理以下能力：

- 基础 URL 与环境区分
- 请求头注入
- 登录态或 Token 处理
- 统一错误提示
- 重试、超时、取消请求
- 响应数据格式标准化

通过中间层封装，可以让业务页面更专注于状态与交互，而不是充斥大量重复的接口样板代码。

### 3. 工程规范落地

建议在仓库中集成以下基础规范：

- `ESLint`：统一代码风格与潜在问题检查
- `Prettier`：统一格式化输出
- `Husky` + `lint-staged`：在提交前自动检查关键文件
- `Commitlint`：规范提交信息，便于版本追踪

这些规范看似基础，但对协作质量影响极大。尤其在多人并行开发时，统一约束能显著降低沟通成本。🧩

### 4. 目录结构建议

可以参考下面的思路进行组织：

```bash
src/
  api/        # 接口封装
  utils/      # 通用工具
  components/ # 公共组件
  pages/      # 页面模块
  hooks/      # 复用逻辑
  config/     # 环境与业务配置
```

结构不一定固定，但应当遵循“职责清晰、边界明确、命名统一”的原则。尤其在业务快速扩张时，清晰结构比短期开发速度更重要。

### 5. 最佳实践建议

- 优先沉淀可复用能力，而非复制粘贴
- 业务逻辑和基础能力分层管理
- 为关键工具提供简单示例和注释
- 发布前增加基础检查清单
- 在迭代中持续回收通用模块，避免仓库失控
